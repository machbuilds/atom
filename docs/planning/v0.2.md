# v0.2 — milestone plan

> Status: planning (locked 2026-05-07).
> Scope: 10 items across 3 waves.
> Per-feature plans: [`distribution.md`](distribution.md), [`stack-presets.md`](stack-presets.md).
> v0.1.2 shipped the install-bug patch (item 1) ahead of this milestone.
>
> **De-scope policy is preauthorized**: any item that hits a design wall
> during build can fall out to v0.2.1 with a written reason, without
> blocking the rest. Wave 3 is the most likely de-scope candidate
> because it's the architectural shift.

## What v0.2 ships

Ten items, sequenced so the cheapest wins land first, the user-facing
features ship in the middle, and the architectural shift (install
location + curl one-liner + auto-update) lands last:

```
Wave 1 — foundation, parallelizable, low risk (~1 day)
  #2  VERSION file + `atom upgrade` subcommand
  #3  `./atom-setup --reinstall` flag (stop-gap for stale globals)
  #4  nucleus schema migration tooling (must precede #13 if it rides)
  #5  live `gh repo create` in wizard Git section
  #8  Migration doc for users on the in-place install model

Wave 2 — user-facing features, parallelizable (~3 days)
  #6  Stack presets — Python/FastAPI, Swift/Vapor, Rust/Axum,
       Go CLI/Cobra, TypeScript library
  #7  speckit-constitution integration (replaces TODO marker)

Wave 3 — distribution architecture, sequential, gated on Wave 1 (~2 days)
  #9   `~/.atom/atom/` install + `atom-setup new <name>`
  #10  `install.sh` curl one-liner + README Quick Start rewrite
  #11  `bin/atom-update-check` + snooze (gstack pattern)
```

Wave 1 → Wave 2 (in parallel) → Wave 3. Items inside a wave can run
in parallel where independent.

## Why this scope

v0.1 shipped feature-complete; v0.1.1 did the architectural refactor
of nucleus + learnings (fully user-owned, `~/.atom/` namespace);
v0.1.2 fixed the fresh-clone install crash. v0.2 is the first release
to add **new user-visible surface area** since launch.

The driving question: what holds adoption back today?

1. **No upgrade story.** Once a user installs atom, they have no
   command to fetch a new release. → #2 (`atom upgrade`).
2. **Wizard's stack question is hollow for non-Next.js users.**
   `atom-setup.md` flags this: "Stack presets shipping today: `nextjs`
   only. Other stacks fall back to the generic scaffold." → #6.
3. **Constitution generation is a TODO marker, not a flow.**
   `atom-setup.md` flags it; user runs `speckit-constitution`
   themselves. → #7.
4. **Git push is two-step.** Wizard records intent, user pushes
   manually. → #5.
5. **Install + dev collide.** The folder a user clones IS the project
   IS the dev source. No clean "user install" location. → #9 + #10.
6. **Stale globals stay stale.** Re-running `./atom-setup` only
   installs missing CLIs; doesn't refresh ones already on PATH. → #3
   as stop-gap; #11 as proper auto-update.
7. **Existing in-place users have no upgrade path.** → #8.

These are evidence-backed by the planning docs themselves and by the
distribution decisions locked in [`distribution.md`](distribution.md).

## Wave 1 — foundation

### #2 `VERSION` file + `atom upgrade` subcommand

**What**: a plain text `VERSION` file at the repo root and an
`atom upgrade` subcommand on the existing top-level `atom` help
dispatcher. Polls the user's installed version against the
`raw.githubusercontent.com/.../main/VERSION` URL, runs `git pull`
under the hood, and re-installs each CLI.

**Why now**: without this, no v0.2 feature can reach existing users.
Single most important missing verb.

**Decisions**:

| Decision | Choice | Why |
|---|---|---|
| Version source of truth | Plain `VERSION` file at repo root | Same pattern as gstack; polled via raw GitHub URL — no API, no auth |
| Upgrade verb | `atom upgrade` (subcommand on `atom` CLI, not a separate binary) | One verb, one place; users already know `atom --help` |
| Upgrade mechanism | `cd $INSTALL_DIR && git pull && for cli in ...; do (cd "$cli" && npm install && npm install -g .); done` | Gold-path `git pull` is a known-safe one-liner; the `npm install` step uses the 0.1.2 fix |
| Where install lives | `~/.atom/atom/` (only one of two supported locations until #9 lands; the other is the legacy in-place clone). `atom upgrade` detects which. | Smooth ramp: works for in-place users today; works for `~/.atom/atom/` users after #9 |
| Failure mode | Print the manual upgrade command on any error; never half-upgrade | Failure should leave the user where they were |
| Version comparison | Plain string compare on the SemVer | No fancy version libs; SemVer string compare works for our cadence |

**Risks**:

- `git pull` against a dirty working tree fails. Detect and refuse
  with a clear message ("commit or stash your changes first").
- Network failure during the version check. Treat as a no-op;
  don't print noise.
- `~/.atom/atom/` doesn't exist for users on the in-place model.
  Fall back to "you appear to be on the in-place install; run
  `git pull` from your atom checkout."

**Done when**: `atom upgrade` on a 0.1.2 install with `0.1.3` published
upstream successfully fetches the new version, re-installs CLIs, and
prints "now on 0.1.3."

### #3 `./atom-setup --reinstall` flag

**What**: a flag on the bash wrapper that forces re-install of every
CLI even when find_global already finds it. Closes the "stale globals"
gap until #11 (auto-update) lands.

**Why**: today the wrapper checks `find_global $cli` and skips install
if found. After #2 ships and users have `atom upgrade`, but before #11
lands, `--reinstall` is the explicit "force fresh install" path the
wrapper needs.

**Decisions**:

| Decision | Choice | Why |
|---|---|---|
| Behavior | Set `need_install=1` and run the install loop unconditionally for every CLI in `CLIS` array | One-line change; reuses the existing loop |
| Help text | Surface in `--help` and the script header comment | Discoverable |
| Combinability | Works with `--bare`, `--minimal`, `--full`, `--target`, `--dry-run` | `--reinstall` is a wrapper-level flag; passes through everything else to the wizard |

**Risks**: minimal; the install loop is already idempotent.

**Done when**: `./atom-setup --reinstall --bare --yes --target /tmp/x`
re-installs all five CLIs even when they're already global.

### #4 nucleus schema migration tooling

**What**: a `nucleus migrate` subcommand and a versioned migration
registry inside `bin/nucleus/src/migrations/`. Walks every JSONL file
under `~/.atom/nucleus/projects/*/learnings.jsonl`, applies pending
migrations in order, bumps `schema_version` per entry, writes back
atomically.

**Why now**: nucleus ships `schema_version: 1` today. Any work that
adds a field (semantic search needs `embedding_id`, future features
will want others) without migration tooling forces a hard break for
existing users. Better to ship the system before the first schema
change. Only blocking if #13 lands (deferred to v0.3), but cheap to
ship now.

**Decisions**:

| Decision | Choice | Why |
|---|---|---|
| Trigger | Auto-run on `nucleus add` / `nucleus search` if `schema_version < latest`, with a one-line "migrating N entries" notice | User shouldn't have to know about it; silent enough to not be noisy |
| Migration registry | Numbered files in `src/migrations/00X-description.js`, each exporting `up(entry) → entry` | Mirrors database migration tools |
| Atomicity | Write to `learnings.jsonl.tmp`, rename on success | Crash-safe |
| Rollback | Out of scope | Forward-only; revert ships as a fixed previous version of nucleus |

**Risks**: existing v1 files have no header line; first migration
adds one. Treat that as the test case for the whole subsystem.
Concurrent `nucleus add` during migration — solve with the same
`flock(2)` strategy already flagged in `nucleus.md` for concurrent
appends.

**Done when**: existing v0.1.1 JSONL files (no header) are correctly
migrated on first run, with a smoke test in `scripts/test-atom-setup.sh`
that seeds an old-format file and asserts the post-migration shape.

### #5 live `gh repo create` in Git section

**What**: the wizard's §10 (Git) actually creates the GitHub repo when
the user picks "create new", instead of recording the intent and
deferring to a manual step.

**Why**: today the user runs `gh repo create <name> --private
--source=. --push` themselves after the wizard finishes — a 20-second
extra step that breaks the "one command, project ready" pitch.

**Decisions**:

| Decision | Choice | Why |
|---|---|---|
| Auth detection | `gh auth status` exit code 0 → proceed; else fall back to manual URL paste | Already the v0.1 pattern for nucleus sync repo creation |
| Repo visibility | Private by default, prompt for public | Most projects start private |
| Default name | Project name from §1 | One less question |
| Description | Pulled from §1 | One less question |
| Push behavior | Initial commit and push happen in §10's writer step, not as a separate confirmation | The user already confirmed at the final summary |
| `gh` not installed | Print the exact retry command | Don't block; don't pretend |

**Risks**: `gh repo create` can fail mid-flow (rate limit, name
collision, org permission). Print the exact retry command, don't
abort the whole wizard. Org-owned repos vs personal — v0.2 ships
personal-only.

**Done when**: wizard runs end-to-end with `gh` available, the user
sees the new repo on GitHub at the URL printed in the cheatsheet,
and the working tree's `origin` points at it.

### #8 Migration doc for in-place users

**What**: a short section in `INSTALL.md` (or a dedicated
`docs/MIGRATING.md` if it warrants one) that walks an existing 0.1.x
user from the in-place install model to the `~/.atom/atom/` model
that #9 introduces.

**Why**: when #9 lands, existing users don't automatically benefit.
They need a one-time "clone fresh to `~/.atom/atom/`, point globals
there, optionally remove old clones" walkthrough.

**Decisions**:

| Decision | Choice | Why |
|---|---|---|
| Format | Markdown checklist + commands | Same shape as existing `INSTALL.md` |
| Auto-migration command | `atom migrate-install` subcommand (one-shot) | Mirrors the `~/.nucleus/` → `~/.atom/nucleus/` precedent |
| Old clone disposition | Leave it alone, don't auto-delete | User's data; their call |
| Tooling globals | Re-link via `npm install -g .` against `~/.atom/atom/bin/<cli>/` | Same install loop as `./atom-setup` |

**Risks**: globals point at the old path until re-installed. Doc
must be explicit that re-install is required.

**Done when**: a 0.1.2 user following the doc ends up with
`~/.atom/atom/` populated, globals pointing there, and old clone
optionally removed.

## Wave 2 — user-facing features

### #6 stack presets

See [`stack-presets.md`](stack-presets.md). Five new presets:
Python/FastAPI, Swift (Vapor), Rust (Axum), Go CLI (Cobra), and a
TypeScript library starter. Each ships scaffolding, an opinionated
Dockerfile (where it makes sense), `.env.example`, `README.md`
template, and at least three seeded learnings.

### #7 speckit-constitution integration

**What**: the wizard's §9 (Constitution) runs `speckit-constitution`
inline at the end of `atom-setup`, generating a starter constitution
based on answers from §1 (project basics) and §2 (stack & deploy).
The TODO marker in the cheatsheet goes away.

**Why**: today the constitution step asks "Generate now?" and on yes,
prints "Run `speckit-constitution` after this finishes." Half the
users will forget. Half won't know what `speckit-constitution` is.
Inlining closes the loop.

**Decisions**:

| Decision | Choice | Why |
|---|---|---|
| Invocation | Spawn `speckit-constitution` as subprocess after writer step, before cheatsheet | Same pattern as `nucleus init`; proven |
| Prompt material | Wizard passes a JSON blob (project name, description, stack, deploy, public/internal) via stdin | speckit reads from stdin if available, falls back to interactive |
| Skip path | If user said no in §9, no subprocess | Honor user choice |
| `speckit-constitution` not installed | Print install hint, do not fail the wizard | Fail-soft, consistent with other §-level fallbacks |
| Output location | `<project>/CONSTITUTION.md` | Standard speckit location |

**Risks**: speckit's CLI evolves — pin the wizard against a known
version range and check pre-flight. The JSON-via-stdin contract
requires a small change to speckit if it doesn't support stdin yet —
verify before scoping; if it needs a speckit PR, this becomes a
2-repo change.

**Done when**: a wizard run with §9 = yes produces `CONSTITUTION.md`
populated from the answers, no second command needed.

## Wave 3 — distribution architecture

See [`distribution.md`](distribution.md) for the full architectural
context. Wave 3 implements that architecture across three items.

### #9 `~/.atom/atom/` install + `atom-setup new <name>`

**What**: separate "atom source" from "the project being bootstrapped."
Atom installs to `~/.atom/atom/` (next to `~/.atom/nucleus/` and
`~/.atom/learnings/`). The wizard learns a new verb: `atom-setup new
<name>` creates a fresh directory, scaffolds into it from
`~/.atom/atom/`, leaves the source untouched.

**Why**: today the folder you clone IS the project IS the dev source.
Re-running `atom-setup` against an existing project risks clobbering
it. Dev edits to one clone become live for the "user install."
Decoupling fixes both.

**Decisions** (from `distribution.md`):

| Decision | Choice | Why |
|---|---|---|
| Source location | `~/.atom/atom/` | Completes the `~/.atom/` namespace |
| New project verb | `atom-setup new <name>` | Verb-noun, explicit; the existing `atom-setup` (no verb) becomes deprecated |
| Old in-place model | Stays supported for one release, deprecated with a notice | Don't break existing users |
| Wizard reads templates from | `$HOME/.atom/atom/` (env-overridable for tests) | Single source of truth |

**Risks**: existing users on the in-place model. #8 (migration doc)
covers the manual path. Wizard must detect and warn when run from
the legacy in-place location.

**Done when**: `atom-setup new my-project` (run anywhere on PATH)
produces `./my-project/` with the scaffold copied from
`~/.atom/atom/`, and `~/.atom/atom/` is unchanged.

### #10 `install.sh` curl one-liner + README Quick Start rewrite

**What**: a one-line install command at the top of the README:

```bash
curl -fsSL https://raw.githubusercontent.com/machbuilds/atom/main/install.sh | bash
```

`install.sh` clones to `~/.atom/atom/`, runs `npm install` then
`npm install -g .` for each CLI, prints "atom installed. Run
`atom-setup new <project-name>` to start your first project."

**Why**: gstack-style install UX. The current `git clone … && cd … &&
./atom-setup` is fine but not memorable. A curl one-liner is the
modern pattern users expect. Now unblocked because the repo is public.

**Decisions**:

| Decision | Choice | Why |
|---|---|---|
| Install location | `~/.atom/atom/` | Consistent with #9 |
| Idempotency | Re-running curl pipe should detect existing install and offer `atom upgrade` | Don't clobber |
| Pre-existing `~/.atom/atom/` | If a git checkout, refuse with a clear "use `atom upgrade`" message | Safety |
| Network failure | Standard `curl` error; no retries | Caller's problem |
| Verification | Optional checksum step; defer to v0.3 if it adds noise | Not blocking; people who care can `git verify-tag v0.1.2` |

**Risks**: `curl | bash` raises eyebrows. Mitigate by also
documenting the manual `git clone` path on the line below. SECURITY.md
should call out the trust model.

**Done when**: a clean macOS or Linux machine with no atom installed
runs the curl one-liner and ends up with `~/.atom/atom/`, all five
globals on PATH, and `atom --help` working.

### #11 `bin/atom-update-check` + snooze

**What**: a `~/.atom/atom/bin/atom-update-check` script that runs
lazily (on the first invocation of any atom CLI per session) and
prints a one-liner if a new version is published upstream. Snoozable
24h / 48h / 7d. State lives in `~/.atom/state/update-check.json`.

**Why**: even with `atom upgrade`, users won't know to run it.
Auto-notification (gstack-pattern) makes upgrades a one-line accept.

**Decisions**:

| Decision | Choice | Why |
|---|---|---|
| Trigger | Lazy on CLI startup (every `atom`, `atom-setup`, `nucleus`, `learnings`, `model-race` invocation) — fork-and-forget, capped at one check per 6h per machine | No background daemon; no shell rc edit |
| Source | `raw.githubusercontent.com/.../main/VERSION` | Same as #2 |
| Throttle | 6h per machine, 24h after notice shown, snoozable to 48h or 7d | Avoid notification fatigue |
| Snooze UX | "atom 0.2.1 is available. (run `atom upgrade`, or snooze: `atom upgrade --snooze 7d`)" | One line; explicit verb |
| Network safe-silent | Failures print nothing; check is best-effort | Don't pollute CLI startup |

**Risks**: forks-on-startup add latency. Cap at 50ms via a child
process with stdout silenced; if the network call hasn't returned
in 50ms, it dies and the user sees nothing. Same shape as gstack.

**Done when**: a 0.1.x install with 0.2.0 published upstream prints
the upgrade notice exactly once on the next CLI invocation, and
`atom upgrade --snooze 7d` suppresses it for a week.

## Risks across the milestone

- **Wave 3 absorbs time.** The architectural shift touches the
  wizard, the bash wrapper, the README, and existing users. The
  preauthorized de-scope is the answer; document the call when it
  triggers, don't delay the milestone debating it.
- **Wave 2 stack presets is L by raw work, not complexity.** Five
  presets is five Dockerfile-tunings, five sets of learnings, five
  `.env.example` files. Parallelize across presets to keep wall time
  down. See `stack-presets.md` for the per-preset checklist.
- **Demo GIF (memory directive)**: not in v0.2 scope. Capturing a
  GIF before the v0.2 wizard changes lands wastes the recording.
  Defer to v1.0 when the wizard surface is final.
- **Schema migration is plumbing**: #4 ships even if no v0.2 item
  consumes it, because the next person to add a field will need it.
- **In-place users left behind**: #8 must land in the same release
  as #9 + #10. Otherwise upgrading is "your atom doesn't work
  anymore, here's how to fix it" instead of a smooth migration.

## Out of scope for v0.2 (v0.3 candidates)

- **`model-race --auto`** — cross-CLI parallel session orchestration.
  Risky (per-CLI adapter contract, three independent CLIs to maintain).
  See [`model-race-auto.md`](model-race-auto.md). v0.3.
- **`nucleus search --semantic`** — local embeddings via
  `@xenova/transformers`. Risky (embedding-model choice, vector
  store, indexing strategy, quality bar against `rg`). v0.3.
- **GitHub stars badge** in README (cosmetic).
- **Org-owned repos** in `gh repo create` (no demand yet).
- **Demo GIF / screencast** for README. v1.0 when the wizard surface
  is final.
- **Tool-specific wrappers** for Codex / Gemini / Cursor (waiting
  for their skill systems to mature).

## Out of scope for v0.3 too

- **Hosted "atom service"** (web UI, central sync). Out of scope
  permanently per the user-owned-data principle.
- **Publishing to the npm registry**. Explicitly rejected; see
  `distribution.md`.
- **Self-update mechanism that bypasses git**. The model is `git pull`
  under the hood; the user-facing verb (`atom upgrade`) just polishes
  it.

## Dependencies

- Wave 1 items (#2, #3, #4, #5, #8) have no internal blockers; can
  ship in any order or in parallel.
- Wave 2 (#6, #7) does not depend on Wave 1.
- Wave 3 (#9, #10, #11) depends on #2 (`atom upgrade` must exist
  before the curl one-liner can recommend it on re-run).
- #8 (migration doc) ships in the same release as #9 (architectural
  shift). Either both or neither.

## Success criteria

- A user on 0.1.2 runs `atom upgrade` and ends up on the latest
  v0.2.x without manual intervention.
- A new user picking Python, Swift, Rust, Go CLI, or "library" in
  the wizard gets a working scaffold with stack-appropriate
  Dockerfile and learnings, not a generic fallback.
- `atom-setup` ends with a created GitHub repo, a pushed initial
  commit, and a generated `CONSTITUTION.md` — no second command.
- A clean macOS install runs the curl one-liner and ends up with
  `~/.atom/atom/` populated, globals on PATH, and `atom-setup new
  <name>` working.
- v0.1.x nucleus data migrates to v0.2 format silently on first
  use, with no data loss.
